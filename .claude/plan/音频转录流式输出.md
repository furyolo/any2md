# 音频转录流式输出改造计划

## 需求概述

将音频转录从批量处理模式改进为流式输出模式，使用户能够实时查看转录进度和部分结果。

## 技术目标

1. **流式写入**：每完成一个切片转录后立即追加写入 markdown 文件
2. **进度显示**：在 CLI 中实时显示转录进度（切片索引、耗时）
3. **断点续传**：保存中间结果，支持失败后恢复（可选，第一版不实现）
4. **向后兼容**：不破坏现有功能和接口

## 架构分析

### 当前数据流

```
音频文件 → 切片 → 逐个转录 → 收集结果 → 一次性返回 → ConversionService写入文件
```

### 目标数据流

```
音频文件 → 切片 → 逐个转录 → 每个切片完成后立即追加写入 → 返回完整路径
```

### 关键修改点

1. **QwenAsrAudioConverter._process_audio_file()** (audio.py:328-350)
   - 接收 `output_path` 参数
   - 每个切片完成后立即追加写入文件
   - 通过 `progress_callback` 发送进度事件

2. **LocalQwenAudioConverter.__call__()** (audio.py:312-326)
   - 接收 `output_path` 参数并传递给 `_process_audio_file()`

3. **ConversionService._execute_job()** (app.py:300-380)
   - 判断 converter 是否支持流式输出
   - 对于支持流式的 converter，提前创建输出文件并传递路径
   - 跳过重复写入文件的逻辑

4. **CLI 进度显示** (cli.py)
   - 实现 `progress_callback` 函数
   - 在 stderr 输出进度信息（不影响 stdout 的 markdown 输出）

### Step 4: 实现 CLI 进度显示

**修改 4.1**：定义进度回调函数

```python
# 在 main() 函数开始处添加
def _create_progress_callback(stderr):
    """创建进度回调函数，用于显示转录进度"""
    def callback(event: dict):
        kind = event.get("kind")
        if kind == "chunk_start":
            idx = event.get("index", 0)
            total = event.get("total", 0)
            print(f"[{idx}/{total}] Starting transcription...", file=stderr)
        elif kind == "chunk_complete":
            idx = event.get("index", 0)
            total = event.get("total", 0)
            elapsed = event.get("elapsed", 0)
            print(f"[{idx}/{total}] Completed in {elapsed:.1f}s", file=stderr)
        elif kind == "chunk_written":
            idx = event.get("index", 0)
            total = event.get("total", 0)
            text_len = event.get("text_length", 0)
            print(f"[{idx}/{total}] Written {text_len} characters to output", file=stderr)
        elif kind == "chunk_retry":
            idx = event.get("index", 0)
            attempt = event.get("attempt", 0)
            print(f"[{idx}] Retry attempt {attempt}...", file=stderr)
    return callback
```

## 技术风险与缓解

### 风险 1：文件写入失败
**缓解**：每次追加写入后立即 flush，捕获 IOError 并保留已完成的结果

### 风险 2：t2s 后处理与流式写入冲突
**缓解**：流式写入原始文本，如果启用 t2s，在所有切片完成后重新读取文件并应用后处理

### 风险 3：AUC 后端不支持流式
**缓解**：仅对 qwen-local 后端启用流式，AUC 保持原逻辑

### 风险 4：progress_callback 在不同转录路径中不一致
**缓解**：统一在 `_transcribe_chunk_with_retry()` 中发送事件

## 验收标准

1. ✅ 用户在转录长音频时能实时看到每个切片的转录结果写入文件
2. ✅ CLI 输出显示清晰的进度信息（如"[3/10] Completed in 45s"）
3. ✅ 如果中途失败，已完成的切片结果已保存到文件
4. ✅ 不影响短音频（单切片）的转录流程
5. ✅ 不影响 AUC 后端的转录流程
6. ✅ 通过现有测试用例（如果有）
